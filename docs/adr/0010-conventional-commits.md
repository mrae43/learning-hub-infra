# 0010 — Conventional Commits, Enforced in CI

## Status
Accepted

## Context
`release.yml` (established in an earlier session) generates a changelog from commit history on merge/tag. Automated changelog generation is only reliable if commit messages follow a parseable convention — otherwise the changelog degrades into a list of inconsistent, hand-written messages, and any future automation (e.g. semantic version bumps inferred from commit type) has nothing structured to key off of.

## Decision
Commits follow the **Conventional Commits** format (`feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`, `ci:`, etc.), with standard git hygiene on top (imperative mood, concise subject line, body wrapped for readability). This is enforced, not just documented as a suggestion — a CI check rejects non-conforming commit messages on PRs, consistent with the existing "merge blocked on failing job" pattern from `tech-stack.md`.

## Consequences
- The changelog generated by `release.yml` is structured and trustworthy — entries can be grouped by type (features vs. fixes vs. chores) without manual curation.
- Commit type prefixes create room for automated semantic versioning later (`feat:` → minor bump, `fix:` → patch, breaking-change markers → major) if that's ever wanted — not required now, but available as a free byproduct of the convention.
- Enforcement adds a small amount of friction to every commit (must remember the prefix, must write a real imperative subject line) — acceptable trade for changelog quality, per the decision to enforce rather than merely suggest.
- Full mechanics (exact commit types allowed, enforcement point — pre-commit hook vs. CI-only) are specified in `commit-instructions.md`.
